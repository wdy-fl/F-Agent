# F-Agent 项目目标与路线图

## 1. 项目定位

F-Agent（阿福）是一个面向个人研发场景的本地 CLI 智能助手。项目吸收 Hermes-Agent 的核心思想，但不照搬其生产级复杂度，而是围绕“可运行、可扩展、可进化”三个目标，逐步构建一个能长期陪伴用户工作的个人 Agent。

当前阶段的重点不是做一个大而全的平台，而是把 Agent 的核心闭环打牢：

- **可运行**：通过 CLI 启动，完成多轮对话、工具调用、会话持久化和错误恢复。
- **可扩展**：工具、记忆、技能、配置、定时任务等能力以独立模块组织，后续可以按需扩展。
- **可进化**：通过记忆、技能和上下文压缩，让 Agent 能沉淀经验，并在长对话和跨会话中持续复用。

## 2. 阶段性目标

### 2.1 短期目标：稳定核心闭环

短期目标是让阿福成为一个稳定可用的本地研发助手，优先保证核心路径可靠：

1. 用户通过 CLI 发起任务。
2. Agent 构建系统提示词，注入技能索引、记忆上下文和工作区指引。
3. LLM 进行推理，按需调用终端、文件、Web、记忆、技能等工具。
4. 工具结果回传给 LLM，继续推理直到给出最终回复。
5. 会话、消息、工具结果、记忆同步和压缩状态被持久化，支持后续恢复和召回。

这个阶段的成功标准是：项目可以被真实使用，而不是只停留在框架或演示代码。

### 2.2 中期目标：形成个人 Agent 能力体系

中期目标是让阿福从“能调用工具的聊天助手”演进为“能沉淀工作经验的个人 Agent”：

- 记忆系统能稳定记录用户偏好、项目背景和跨会话上下文。
- 技能系统能把高价值工作流沉淀为可复用能力，并通过渐进式披露降低上下文成本。
- 工具系统在保持简单的前提下，具备更清晰的安全边界和扩展方式。
- 文档、测试和代码实现保持一致，方便持续迭代。

### 2.3 长期目标：探索多端与自治能力

长期目标是在核心闭环稳定后，逐步探索更强的 Agent 形态：

- 多平台入口：CLI、Web、消息平台等多端统一接入。
- 子 Agent 委托：将独立任务拆分给隔离上下文的子 Agent 执行。
- 定时任务增强：在 CLI 常驻调度基础上，继续探索独立守护进程、补跑策略、任务编辑和多端通知。
- MCP / 插件生态：接入外部工具协议和更丰富的能力市场。
- Web Dashboard / IDE 集成：提供更直观的会话、记忆、技能和任务管理界面。

这些能力属于路线图，不代表当前版本已经完整实现。

## 3. 当前能力快照

### 3.1 已落地能力

| 能力 | 当前状态 |
|------|----------|
| Agent 主循环 | 已实现 `agent/loop.py`，支持多轮 LLM 调用、工具执行、预算控制和最终回复 |
| LLM 接入 | 已通过 OpenAI SDK 封装 `llm/client.py`，支持 `base_url` 切换兼容模型 |
| CLI 交互 | 已实现 `cli/interface.py` 和 `main.py`，结合 prompt_toolkit 与 rich 提供本地交互 |
| 工具注册与调度 | 已实现 `tools/registry.py`，采用显式注册和按顺序串行调度 |
| 文件与终端工具 | 已实现终端执行、文件读写、目录查看等基础工具 |
| Web 工具 | 已实现 Web 搜索 / 抓取相关工具能力 |
| 会话持久化 | 已实现 SQLite 会话、消息、统计信息和会话恢复 |
| 全文搜索召回 | 已基于 SQLite FTS5 建立消息索引，用于历史消息检索 |
| 记忆系统 | 已实现 `memory/manager.py`、`MEMORY.md`、`USER.md`、`SOUL.md`、`AGENT.md` 等持久化文件读写与同步 |

| 上下文压缩 | 已实现工具结果裁剪、结构化摘要、头尾保护和压缩状态恢复 |

| 技能管理工具 | 已实现 skills_list、skill_view、skill_manage 等工具入口 |

| 命令审批 | 已实现命令审批配置与 CLI 审批回调，用于拦截高风险终端命令 |
| 定时任务 | 已实现 CLI 常驻期间的定时 Agent prompt 调度，支持 create/list/pause/resume/remove、SQLite 持久化和任务级危险命令授权 |
| 数据库工具 | 已实现 MySQL 只读查询工具配置与调用入口 |
| 工程测试 | 已建立覆盖 Agent Loop、工具、记忆、压缩、技能、定时任务、CLI 等模块的 pytest 测试 |
| 启动与配置 | 已提供 `config.yaml.example`、`start.sh`、`start.ps1` 和 pyproject 配置 |

### 3.2 正在完善的能力

| 能力 | 需要完善的方向 |
|------|----------------|
| 文档一致性 | README、GOAL、ARCHITECTURE 中部分能力描述仍偏路线图，需要持续校准到真实实现 |
| 用户画像表达 | 当前用户画像由 `memory/manager.py` 统一维护，相关文档需统一到这一实现口径 |
| 技能生命周期 | 技能创建、安装、查看、编辑已有基础能力，但更新、卸载、版本管理和使用体验仍可增强 |
| 记忆质量 | 已有自动提取与持久化机制，但还需要继续优化触发时机、去重策略和长期可控性 |
| 工具安全边界 | 已有命令审批、定时任务任务级授权和 Skills Hub 防护，但终端、网络、数据库等工具仍需要更系统的权限模型 |
| 定时任务体验 | 已有 CLI 常驻调度和管理工具，后续可增强任务编辑、立即运行、错过任务处理和通知查看体验 |
| 会话体验 | 会话恢复、标题、统计已具备基础能力，后续可增强检索、列表、标签和上下文选择体验 |
| 上下文压缩稳定性 | 已支持迭代摘要和工具结果裁剪，后续继续验证更多 OpenAI tool call 边界情况 |

### 3.3 尚未实现或仅处于路线图的能力

以下能力不应被描述为当前已完成：

- 多平台消息网关（Telegram / Discord / Web 等）。
- 子 Agent 派生、并行委托和结果汇总。
- 定时任务的独立守护进程、任务更新、立即运行、补跑和多端通知。
- MCP 协议接入。
- Web Dashboard。
- IDE 集成。
- Docker / SSH / 云沙箱等多环境执行后端。
- 语音交互和 RL 训练环境。

## 4. 路线图

### 4.1 近期路线：校准与稳定

- 持续修正文档中超前于实现的表述，保证能力声明可信。
- 补齐关键模块 README 与测试说明，让新模块能独立理解和验证。
- 优化上下文压缩、记忆同步、会话恢复等核心路径的边界情况。
- 保持测试套件稳定，确保每次迭代都能自测核心闭环。

### 4.2 中期路线：记忆、技能与安全

- 优化记忆提取策略：减少噪声、重复和过期信息。
- 增强技能系统：支持更清晰的安装来源、版本记录、更新策略和卸载体验。
- 扩展工具权限模型：让终端、网络、数据库等工具具备更细粒度的审批和配置。
- 将高频研发流程沉淀为可复用技能，形成阿福自己的能力库。

### 4.3 长期路线：平台化探索

- 在 CLI 稳定后，再评估 Web Dashboard 和消息平台接入。
- 在工具和会话隔离成熟后，再引入子 Agent 委托与并行执行。
- 在 CLI 常驻定时任务稳定后，再评估独立守护进程、任务更新、立即运行、补跑和多端通知。
- 在插件边界清晰后，再接入 MCP 和更通用的外部能力生态。

## 5. 设计原则

1. **能力声明匹配实现状态**  
   文档必须区分“已实现”“正在完善”“路线图”，避免把未来目标写成当前能力。

2. **核心闭环优先**  
   优先保证 Agent Loop、工具调用、记忆、技能、上下文压缩这些核心路径可运行、可测试、可恢复。

3. **保持简洁**  
   不为了复刻 Hermes-Agent 而引入生产级复杂度。能用清晰模块解决的问题，不提前抽象成平台。

4. **知识靠近代码**  
   全局目标写在 `GOAL.md`，架构写在 `ARCHITECTURE.md`，模块说明写在对应模块目录，具体行为尽量靠近实现和测试。

5. **安全默认保守**  
   终端、网络、数据库、外部技能安装等能力默认保持可审计、可审批、可回退。

6. **渐进式演进**  
   每个阶段都交付可使用的系统，再在真实使用中决定下一步扩展，而不是一次性设计完整平台。
