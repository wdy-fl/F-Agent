# Agent 开发 SOP 设计文档

> 指导 Ducc（Claude Code）在 F-Agent 项目开发中遵循的工作流规范

## 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 组织形式 | 分层式（核心原则 + 环节规则） | 适配柔性 SOP 定位，原则提供方向感，规则提供具体建议 |
| 存储位置 | `docs/dev-sop.md` | 拆分独立文件，符合渐进式披露原则 |
| 约束力度 | 柔性 | 每个环节提供指导原则和建议清单，Agent 根据任务灵活判断 |
| 进度持久化 | `docs/plans/` 目录下的进度文档 | 跨会话可接续，符合 AICD 持久性原则 |

## 核心原则（5 条）

### 1. 先对齐再动手

在写任何代码之前，先与用户确认需求理解一致。需求模糊时必须主动澄清，不应自行脑补。

**触发点**：用户提出新需求时
**覆盖环节**：需求对齐 → 任务规划

### 2. 先理解再规划

动手前先探索相关代码、文档、已有实现，理解现状后再出方案。避免在信息不足时拍脑袋设计。

**触发点**：进入项目理解/任务规划环节时
**覆盖环节**：项目理解 → 任务规划

### 3. 小步前进，步步验证

每个改动要足够小，能在一次验证中确认正确性。写完就测，不积压问题。

**触发点**：代码编写过程中
**覆盖环节**：代码编写 → 自测检查

### 4. 进度可视，及时同步

每完成一个任务点，更新进度状态，让用户始终知道当前在哪里、下一步做什么。

**触发点**：任务点完成时
**覆盖环节**：进度设置 → 进度更新 → 汇报总结

### 5. 原子提交，可追溯

每次 commit 对应一个明确的、可回滚的变更单元。提交前确保文档和测试已同步。

**触发点**：任务点完成且自测通过时
**覆盖环节**：Git 提交

## 环节规则

### 1. 需求对齐

当用户提出新需求时：

- [ ] 用自己的话复述一遍需求，让用户确认理解是否正确
- [ ] 识别需求中的模糊点，主动提问澄清（每次一个问题，不轰炸）
- [ ] 如果需求涉及多个独立子系统，先拆分再逐个讨论
- [ ] 明确输入/输出/成功标准，确认"做到什么程度算完成"

**何时跳过**：用户指令已经非常具体、无歧义（如"把第 3 行的 `port` 改成 `8080`"）

### 2. 项目理解

进入实现前：

- [ ] 阅读 CLAUDE.md 和相关文档路由指向的文档
- [ ] 检查是否已有该方向的进度文档（如 `docs/plans/phase1-mvp.md`），如果有则先读取，了解已完成的开发状态和停滞点
- [ ] 用 Glob/Grep 找到相关代码文件，理解现有结构和命名风格
- [ ] 检查 git log 了解最近的变更方向
- [ ] 如果涉及外部依赖或参考项目，先读参考代码理清核心逻辑

**何时跳过**：任务不涉及现有代码（如新建独立文件）

### 3. 任务规划

动手前的最后一步：

- [ ] 将需求分解为具体的任务点，用 TaskCreate 工具记录
- [ ] 创建进度规划文档（放在 `docs/plans/` 目录），格式包含：任务列表、当前状态、完成标准
- [ ] 向用户呈现任务列表，获得确认后才开始执行
- [ ] 大任务（>3 个任务点）优先考虑使用 EnterPlanMode 做详细设计

### 4. 进度设置

任务规划确认后：

- [ ] 在进度文档中初始化每个任务点的状态（待开始/进行中/已完成）
- [ ] 标注任务点之间的依赖关系和执行顺序
- [ ] 将任务点同步到 TaskCreate 工具中（会话内的即时跟踪）
- [ ] 让用户知道第一个任务点是什么、预计动哪些文件

### 5. 代码编写

执行每个任务点时：

- [ ] 先读后改：改文件前先 Read 确认当前内容，不做盲改
- [ ] 保持文件小巧：单个文件尽量不超过 300 行，超过时考虑拆分
- [ ] 遵循已有模式：命名、结构、错误处理方式与现有代码一致
- [ ] 不做过度设计：不写"以后可能会用到"的代码，不引入任务范围外的重构
- [ ] 安全第一：不引入命令注入、SQL 注入、XSS 等安全漏洞
- [ ] 函数有描述：每个函数需要描述其用途，复杂逻辑需要注释说明意图
- [ ] 日志与异常：关键处理环节记录日志，异常捕获信息要包含足够的上下文（堆栈、参数、状态），确保能定位和修复问题

### 6. 自测检查

每个任务点写完代码后：

- [ ] 运行相关测试（如果有测试套件）
- [ ] 对功能代码手动验证关键路径（如 `python3 main.py` 启动确认）
- [ ] 检查是否有 lint 错误或类型问题
- [ ] 如果涉及 CLI，实际执行一遍确认输出符合预期
- [ ] 确认未破坏已有功能（回归检查）

### 7. 进度更新

每个任务点完成并自测通过后：

- [ ] 在进度文档中将该任务点标记为已完成，记录完成时间和关键产出
- [ ] 在会话内用 TaskUpdate 同步更新任务状态
- [ ] 主动告知用户当前进度（已完成 N/M），询问是否继续下一个

### 8. 汇报总结

当用户要求汇报或阶段性里程碑达成时：

- [ ] 对照进度文档，总结：已完成项、进行中项、待开始项
- [ ] 列出本次会话的关键产出（新增/修改了哪些文件，解决了什么问题）
- [ ] 标记遇到的阻塞点或需要用户决策的事项
- [ ] 如果所有任务点完成，给出收尾建议（如是否需要 git commit、是否进入下一个阶段）

### 9. Git 提交

用户确认要提交时：

- [ ] 检查是否所有改动文件都是有意义的（无无关的调试代码、临时文件）
- [ ] 提交信息聚焦"为什么改"，而非"改了什么"
- [ ] 单次提交对应一个原子变更，不把多个独立改动混在一起
- [ ] 进度文档和测试用例已同步更新（AICD 一致性原则）
- [ ] 提交后确认 git status 干净
