## GAP-01：文件存储位置不一致

**状态**：待修文档  
**优先级**：中

### 架构文档声明

`docs/ARCHITECTURE.md` 中描述配置、数据库、用户画像、技能和日志位于用户级目录 `~/.fagent/`：

- 用户画像：`~/.fagent/USER.md`
- 配置：`~/.fagent/config.yaml`
- 数据库：`~/.fagent/state.db`
- 技能：`~/.fagent/skills/`
- 日志：`~/.fagent/logs/agent.log`

### 代码现状

当前实现使用项目内 `workspace/`：

- `workspace/config.yaml`
- `workspace/state.db`
- `workspace/USER.md`
- `workspace/skills/`
- `workspace/logs/`

对应实现位于 `config/settings.py` 的默认路径常量。

### 影响

用户和开发者可能根据架构文档去 `~/.fagent/` 查找配置或记忆，但真实文件在项目内 `workspace/`。

### 待讨论问题

是否将架构文档改为当前的 `workspace/` 方案，还是将代码迁移为 `~/.fagent/` 方案？

### 讨论结论

保留当前项目内 `workspace/` 存储方案，同步架构文档中的文件存储位置；不迁移代码到 `~/.fagent/`。

## GAP-02：预算控制 grace call 可能多调用一次

**状态**：已关闭  
**优先级**：高

### 架构文档声明

预算耗尽后允许一次额外调用（grace call），用于让 LLM 输出最终回复。

### 代码现状

`IterationBudget.can_continue()` 在预算为 0 时会额外返回一次 `True`，主循环会继续执行一次正常 LLM 调用。主循环跳出后又调用 `_grace_call()`。

这可能导致实际调用次数为 `max_iterations + 2`，而不是文档期望的 `max_iterations + 1`。

### 影响

可能带来额外 token 消耗，并让预算语义变得不清晰。该问题更像代码行为缺陷，而不是单纯文档偏差。

### 待讨论问题

是否修改预算控制实现，使 grace call 只发生一次，并明确由 `AgentLoop._grace_call()` 负责？

### 讨论结论

修代码，保留架构语义：最大正常迭代为 `max_iterations` 次，预算耗尽后最多只允许一次 grace call；grace call 由 `AgentLoop._grace_call()` 统一负责，`IterationBudget.can_continue()` 不应额外放行一次正常 LLM 调用。

实施结果：IterationBudget.can_continue() 只判断正常预算，唯一额外调用由 AgentLoop._grace_call() 负责。

## GAP-03：混合工具调用时执行顺序可能改变

**状态**：已关闭  
**优先级**：高

### 架构文档声明

只读工具可并行执行，有副作用工具顺序执行。

### 代码现状

`ToolRegistry.dispatch_parallel()` 会先收集所有并行安全工具并执行，再顺序执行所有不安全工具，最后按原始 tool_calls 顺序组装结果。

结果顺序被保留，但执行顺序可能改变。例如 LLM 返回“先写文件，再读文件”，当前实现可能先读后写。

### 影响

如果 tool_calls 之间存在隐含依赖，执行顺序改变可能导致结果错误。

### 待讨论问题

是否保留工具并行能力，还是所有工具调用都按 LLM 返回顺序串行执行？

### 讨论结论

不需要工具并行。修代码为所有工具调用都按 LLM 返回顺序串行执行，确保执行顺序与模型规划顺序一致；同步架构文档，移除默认并行执行和只读工具并行的设计表述。

实施结果：ToolRegistry.dispatch_parallel() 保留历史方法名但改为严格串行执行，结果顺序与调用顺序一致。

## GAP-04：上下文压缩算法比架构文档简化

**状态**：已关闭  
**优先级**：中

### 架构文档声明

压缩算法包括：

1. 先裁剪旧工具结果。
2. 保护 head 和 tail。
3. LLM 生成结构化摘要。
4. 后续压缩基于旧摘要和新对话迭代更新。
5. 连续两次压缩节省小于 10% 时跳过。

### 代码现状

当前 `ContextCompressor.compress()` 主要执行 head/middle/tail 切分，然后对 middle 生成摘要。

`trim_tool_results()` 已实现，但没有在 `compress()` 中调用。当前也没有显式实现“基于旧摘要迭代更新”的逻辑。

### 影响

当前实现能完成基础长对话压缩，但不是架构文档描述的完整版算法。

### 待讨论问题

是把架构文档改成当前简化实现，还是继续补全工具结果预裁剪和摘要迭代更新？

### 讨论结论

实现架构文档描述的完整上下文压缩算法，不降级修改文档。后续需补齐：压缩前接入旧工具结果裁剪、保护 head/tail、基于旧摘要和新增对话进行迭代更新、保留反抖动策略，并确保压缩后消息格式可继续被 LLM 和工具调用流程使用。

实施结果：压缩流程已接入工具结果裁剪，并支持基于旧摘要和新增对话的迭代摘要更新。

同步架构文档职责描述：AgentLoop 负责消息持久化，FTS5 由 DB 触发器维护；MemoryManager.sync 仅作为记忆系统的扩展钩子，不承担写入 SQLite 的职责。

## GAP-09：用户画像存在直接覆盖和 LLM 合并两条路径

**状态**：待修文档  
**优先级**：低

### 架构文档声明

LLM 通过 memory 工具调用 `update_profile`，工具读取当前 USER.md 和新观察，通过 LLM 合并后写回。

### 代码现状

memory 工具有两种画像写入路径：

- `save`：直接写入或覆盖画像内容。
- `update_profile`：通过 `UserProfileManager.update_profile()` 调用 LLM 合并画像。

### 影响

直接 `save` 可能被误用为覆盖画像；文档没有说明这条路径的语义和风险。

### 待讨论问题

是否保留 `save` 动作？如果保留，是否应在文档中明确它是“直接覆盖/保存”，并建议默认使用 `update_profile`？

### 讨论结论

保留 `save` 和 `update_profile` 两条工具动作；同步架构文档说明：常规画像更新优先使用 `update_profile` 进行 LLM 合并，`save` 是直接保存/覆盖画像内容的低级能力。
